# DEMO TALK TRACK

## 30 秒介紹

這是一個銀行 AI 客服的概念驗證 demo。我把流程拆成三個最小能力：先做 rule-based 風險分流，低風險問題再進 LangChain RAG，最後把每次互動寫成簡單 audit log。這樣可以展示 AI 能力，同時保留銀行情境需要的保守性與可追蹤性。

## 90 秒介紹

這個 PoC 的目標不是模擬完整銀行系統，而是做一個明天面試可穩定展示的 AI 客服 demo。整體流程很簡單：使用者先輸入問題，router 會先判斷風險等級。像轉帳、個資、申訴、額度調整這種高風險情境，系統不會自動回答，而是直接要求人工處理或身份驗證。若是 FAQ、產品介紹、流程查詢等低風險問題，才會進入 LangChain 的最小 RAG 流程，從本地 FAQ 文件中檢索內容，再產生一個保守回答，並把引用片段顯示在畫面上。整個流程還會同步產生 audit log，記錄使用者問題、路由結果、是否有檢索到文件，以及回覆類型。為了降低 demo 風險，這個專案支援兩種模式：有 OpenAI-compatible endpoint 時走 live mode，沒有時就進 mock mode，仍然可以完整展示整體設計。

## 5 個可能追問與建議回答

### 1. 為什麼高風險問題不直接讓模型回答？

因為銀行場景的風險容忍度很低。像轉帳、個資、申訴或額度調整，都不適合只靠一般對話模型直接回答或執行。這個 PoC 刻意把高風險問題擋在前面，強調安全邊界。

### 2. 這個 RAG 為什麼只用本地 FAQ？

因為這次目標是做可攜、可重現的面試 demo。本地 FAQ 搭配 in-memory retrieval 可以降低依賴，避免資料庫、網路或權限問題影響展示。

### 3. 如果 embedding 掛掉怎麼辦？

系統會先 fallback 到本地 deterministic embedding，只有在本地 embedding 也不足時，才再退到 keyword retrieval。這是刻意設計的 resilience 機制，讓整個 demo 不會因為外部 embedding 服務失敗而完全不能用。

### 4. 為什麼回答要附來源？

因為銀行情境裡，回答可追溯比回答華麗更重要。附來源片段可以讓面試官看到這不是純生成，而是有檢索依據的保守回答。

### 5. 這個 PoC 之後怎麼升級成 production？

可以把本地知識庫替換成 Postgres + pgvector 或正式向量資料庫，把 audit log 寫入正式稽核系統，並加入身份驗證、權限控管、觀測與人工審核流程。

## 為什麼此 PoC 沒有直接使用 LangGraph，而是先用 LangChain + rule-based router

因為目前流程還是單純的線性流程，沒有複雜的多節點狀態管理需求。這次的主要目的，是把「風險分流 + RAG + audit log」做成一個短時間內可穩定展示的最小系統。用 LangChain 已經足夠完成這個需求，而且程式碼更少、學習成本更低、面試時更容易講清楚。如果未來流程變成多步驟審核、人工節點、工具切換、長狀態管理，再引入 LangGraph 會更合理。

## 為什麼此 PoC 沒有使用真正資料庫，而是刻意簡化成 in-memory retrieval

因為這次 PoC 的核心不是資料基礎設施，而是展示設計能力與風險控制思維。若一開始就接資料庫，會把 demo 風險拉高，也會讓展示焦點偏離。採用本地 markdown 文件與 in-memory retrieval，可以讓面試官更聚焦在架構取捨：什麼該自動化、什麼不該自動化、什麼需要保守拒答。等 PoC 被認可後，再把知識庫與稽核層替換成正式基礎設施即可。
